RecipeRemix

AI-powered recipe recommendations based on preferences, restrictions, and cuisine fusion.

Overview

RecipeRemix is not just a standard recipe finder but a computational creative AI system that generates novel yet cookable fusion-remix recipes from user-provided ingredients and preferences such as allergies, dietary restrictions, and time limits. The system grounds creativity in real recipe patterns through retrieval-augmented generation (RAG) and improves feasibility through a preference-aware validation and sanity-check layer before giving the recipe to you. By adding a blueprint intermediary layer, constraint-aware validation, iterative refinement through feedback, vision-based ingredient extraction from fridge photographs, and AI-generated food photography, the system expands upon traditional retrieval-augmented generation.

Motivation

Coming up with good recipes can take a lot of time, and sometimes I cannot think of anything to cook or lack the ingredients. This system generates creative recipe remixes based on users’ preferences and ingredients, which helps them a lot with preparation and cooking. Moreover, this will not be generated blindly. First, it will extract common patterns and pertinent genuine recipes from a dataset using retrieval-augmented generation. Then it will produce a structured mix recipe and check on preferences and logic before finalizing the recipe.

Architecture

Single-file Streamlit app (app.py, ~2,000 lines) with a companion offline data-collection script (fetch_spoonacular.py). No backend server, no database — everything runs in-process with CSV as the datastore and Streamlit session_state as the only persistence layer.

Tech Stack

Frontend / App Framework

  • Streamlit — web UI (app.py): sidebar inputs, st.container() layout, st.session_state for cross-rerun state, st.cache_data/st.cache_resource for memoization

AI / ML

  • OpenAI API (openai SDK) — gpt-4o-mini for RAG recipe generation & refinement
  • GPT-4o (vision) — fridge/pantry photo ingredient extraction
  • DALL·E 3 — dish photography generation
  • scikit-learn — TfidfVectorizer + cosine_similarity for recipe retrieval

Data

  • pandas, numpy — data handling
  • CSV (recipes_cleaned.csv) — 17 cuisines × ~8 recipes, sourced via Spoonacular API
  • requests — offline data collection (fetch_spoonacular.py)

Other & Config

  • Pillow (PIL) — image handling, base64 encoding for vision input
  • python-dotenv — env var / secrets loading
  • .env + .streamlit/secrets.toml — API key config (OpenAI, Spoonacular)
  • Local venv — environment

Pipeline

The core algorithm app.py:995–1930

  1. Dataset

    recipes_cleaned.csv — 17 cuisines × ~8 recipes each, cleaned and cached ahead of time (see Data).

  2. Fetch the dataset fetch_spoonacular.py

    Offline, one-time collection script: pulls recipes from the Spoonacular complexSearch/recipes/{id}/information endpoints, cached locally in spoonacular_cache.json to avoid redundant API calls, with a title-keyword override table to fix cuisine mislabeling.

  3. User input

    Ingredients (typed or extracted from an uploaded fridge/pantry photo via GPT-4o vision), dietary preference, allergy, cuisine picks, and max cooking time, collected through the Streamlit sidebar.

  4. Ingredient normalization parse_user_ingredients

    User text or vision-extracted list → parse_user_ingredients → normalized via NORMALIZATION_MAP (e.g. “tomatoes”→“tomato”) and STOP_INGREDIENT_WORDS stripped (units, prep adjectives).

  5. Retrieval retrieve_recipes, app.py:995

    Hybrid scoring: TF-IDF (1–2 grams, English stopwords) cosine similarity between query (ingredients+cuisines+diet) and each recipe’s concatenated search_text, plus a rule-based score (+0.9 per ingredient overlap, −0.1 per missing item, −100 allergy violation, −50 diet violation, −5 over time limit, +2 cuisine match). Recipes with a missing/“unknown” cuisine tag are first re-tagged by infer_cuisine_from_text (app.py:705) — keyword scoring against 17 cuisines’ keyword lists plus a bonus-weighted “unique marker” dictionary (e.g. “gochujang”→Korean +3). Hard filters applied sequentially — allergy → diet → time → cuisine → ingredient-overlap — with graceful fallback to the unfiltered set if a filter empties results. Sorted by overlap → final_score → similarity → time; top-5 returned.

  6. Blueprint build_blueprint, app.py:1084

    A structured intermediate representation (not raw LLM output) combining: dish type (rule-based from DISH_TYPE_RULES), core ingredients (user + pantry extras from CUISINE_PANTRY), cooking methods extracted from retrieved recipes’ instruction text via keyword matching, flavor profile (spicy/bright/creamy/savory heuristics), and estimated time (median of retrieved recipes, capped at the user’s max).

  7. RAG-grounded generation generate_recipe_remix, app.py:1172

    The blueprint + top-3 retrieved recipes (truncated ingredient/instruction text) are injected into a structured prompt to GPT-4o-mini, constrained to strict JSON output; falls back to a rule-based template generator (template_fallback) if the API call fails or returns malformed JSON. The result then passes through a validation & self-correction check (validate_recipe / revise_recipe_from_report, app.py:1349) — ingredient usage, allergy/diet safety, time, step count, and cuisine coherence against CUISINE_KEYWORDS; on failure, it deterministically patches in missing cuisine pantry items and re-validates, no LLM re-call needed.

  8. Image generation generate_recipe_image, app.py:1452

    Builds a descriptive prompt from the recipe + blueprint (plating style chosen by dish type), calls DALL·E 3, and downloads the result via urllib.

  9. Display full recipe

    The validated recipe, its AI-generated photo, ingredients, and numbered steps render in the main Streamlit panel.

  10. Iterative refinement generate_refined_recipe, app.py:1604

    Spice/complexity delta sliders (−2..+2) and a cuisine-swap selector are converted to natural-language instructions and re-sent to GPT-4o-mini with the current recipe + hard constraints, producing a new validated recipe and image.

Data

recipes_cleaned.csv — 17 cuisines × ~8 recipes each, scraped via fetch_spoonacular.py from the Spoonacular complexSearch/recipes/{id}/information endpoints, with a local JSON cache (spoonacular_cache.json) to avoid redundant API calls, and a title-keyword override table to fix cuisine mislabeling.
